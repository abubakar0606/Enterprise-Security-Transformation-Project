# Day 08 - Sysmon Deployment and Advanced Windows Security Monitoring

## Enterprise Security Transformation Project (ESTP)

Day 08 of the Enterprise Security Transformation Project focused on improving endpoint security visibility by integrating Sysmon telemetry with the Wazuh SIEM platform.

Sysmon was used to provide detailed Windows system activity and process-level telemetry from the SecureTech Solutions Domain Controller.

The collected events were forwarded through the Wazuh Agent to the centralized Wazuh Manager for security analysis.

Using the Wazuh Threat Hunting module, detailed process activity, PowerShell execution, discovery activity, command-line information, file-related events, and Windows security events were reviewed.

This phase improved the existing SIEM monitoring architecture by providing deeper endpoint visibility for SOC investigations.

## PowerShell Process Monitoring

Multiple PowerShell process creation events were detected by Wazuh.

The observed rule was:

`Rule ID: 92205`

The rule level was:

`Level 9`

The rule description indicated PowerShell process creation activity.

PowerShell is a legitimate Windows administration tool.

System administrators commonly use PowerShell for:

- Server administration
- Active Directory management
- Service management
- Security configuration
- System automation
- Script execution

However, attackers can also abuse PowerShell.

Possible malicious uses include:

- Payload execution
- Malware execution
- System discovery
- Credential attacks
- File downloads
- Persistence
- Defense evasion
- Remote command execution

Because of this dual-use nature, PowerShell activity should be investigated using event context.

`SecEdit.exe` is a legitimate Microsoft Windows security configuration tool.

It can be used to manage and analyze Windows security policies.

Common legitimate uses include:

- Exporting security policies
- Importing security policies
- Configuring security settings
- Analyzing security configuration
- Applying security templates

The observed command exported security policy information to:

`C:\WINDOWS\TEMP\secpol.cfg`

From a SOC perspective, this activity should be reviewed in context.

## Command-Line Analysis

Command-line information is one of the most valuable fields during endpoint investigations.

The event details displayed:

`data.win.eventdata.commandLine`

The command line provides information about how a process was executed.

For example:

`SecEdit.exe /export /cfg C:\WINDOWS\TEMP\secpol.cfg`

The command contains several important elements.

### Executable

`SecEdit.exe`

Identifies the Windows Security Configuration Editor command-line tool.

### Action

`/export`

Indicates that security configuration information was being exported.

### Configuration Option

`/cfg`

Specifies the configuration file.

### Target File

`C:\WINDOWS\TEMP\secpol.cfg`

Identifies the destination file used by the process.

Command-line analysis helps SOC analysts understand the purpose of process execution.

## File Hash Analysis

The detailed security event included executable hash information.

Hash values can include:

- MD5
- SHA1
- SHA256

Hashes are useful during security investigations.

SOC analysts can use file hashes to:

- Identify known malware
- Search threat intelligence platforms
- Compare files across endpoints
- Create Indicators of Compromise
- Correlate related security events
- Support malware investigations

SHA256 is commonly preferred for security investigations because it provides stronger file identification than MD5.

A file hash alone does not determine whether a file is malicious.

The hash should be compared with trusted threat intelligence sources and internal security data.

- Agent name
